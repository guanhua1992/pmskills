# 评审手册（Review Playbook）

用于准备 PRD 评审材料，防止评审沦为无结构的讨论。

## 评审类型

### 版本评审（Version Review）

目的：确认这个版本为何应当存在。

检查：

- 价值与紧迫性；
- MVP 与版本边界；
- 指标与护栏；
- 依赖与发布风险。

### PRD 内部评审（PRD Internal Review）

目的：在工程评审之前抓出产品逻辑缺口。

检查：

- 需求来源与证据；
- 场景完整性；
- 业务规则；
- 边缘情形；
- 术语一致性；
- 范围与非目标。

### 技术评审（Technical Review）

目的：暴露可行性、集成、数据、安全、性能与迁移风险。

检查：

- 系统边界；
- API 与集成；
- 数据模型与生命周期；
- 权限与审计；
- NFR；
- 灰度与回滚。

### 公开评审（Public Review）

目的：让干系人在交付、上线、运维与验收上对齐。

检查：

- 产品价值与范围；
- 交付计划；
- 验收标准；
- 风险与归属；
- 待决策项。

## 评审文件契约

每个评审文件必须包含：

- 评审对象（Review object）
- 评审人（Reviewers）
- 通过条件（Pass condition）
- 发现（Findings）
- 行动项（Action items）
- 阻断决策（Blocking decision）

若某评审不适用，说明原因。不要留下空白的评审文件。

## 推进规则

`approved` 要求：

- 确定性质量门禁通过；
- 评审文件中不含阻断决策；
- 未解决的评审行动项为非阻断，或已显式指派负责人；
- 用户明确批准该 PRD。
