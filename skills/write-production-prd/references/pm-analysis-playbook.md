# 产品经理分析手册（Product Manager Analysis Playbook）

在编写 PRD 模块之前的需求塑形阶段使用。目标是把一个请求转化为有理有据的产品决策，而不是为一个已经定好的方案做装饰。

## 需求分析序列

1. **需求来源**：谁提出的、什么触发的、有什么证据，以及它是用户需求、业务需求、技术需求、合规需求还是运营需求。
2. **背景与现状**：今天是怎么运转的、谁受影响、流程在哪里断裂、有什么替代方案（workaround）。
3. **目标受众**：主要用户、次要用户、决策者、操作者、支持角色，以及被排除的用户。
4. **价值判断**：用户价值、业务价值、战略契合、紧迫性、影响范围与成本/风险等级。
5. **数据与证据**：使用数据、工单、访谈、截图、日志、竞品案例或代码事实。
6. **业务规则**：状态、权限、计算公式、限额、审批、数据生命周期、初始化与异常处理。
7. **场景与功能映射**：把每个场景映射到任务、流程、功能需求与验收标准。
8. **版本规划**：MVP、v1、未来版本、非目标、依赖、灰度与评审路径。

## 需求质量问题

只问那些影响下一步决策的问题：

- 这是一个现在值得解决的问题吗？
- 谁有痛点，在什么场景下？
- 有什么证据证明痛点是真实的？
- 什么样的可量化结果能证明成功？
- 如果什么都不做会怎样？
- 哪条业务规则若不写，工程就只能靠猜？
- 哪个用户、角色、状态或数据情形最容易被遗漏？
- 什么应当被显式划在范围之外？

## 优先级启发式

用 `P0/P1/P2` 加一句简短理由：

- **P0**：阻断核心价值、合规、安全、数据正确性或上线。
- **P1**：实质提升完成率、效率、质量或可维护性。
- **P2**：有用但可延后，不延后也不会破坏核心场景。

当优先级有争议时，比较价值、受影响用户、频率、风险降低、实现复杂度与可逆性。

## MVP 边界

MVP 不是最小的 UI，而是最小的、连贯的产品行为，它能：

- 端到端解决主要场景；
- 可被测试与衡量；
- 避免不可逆的架构或数据错误；
- 暴露足够的行为以获得真实反馈；
- 留下显式的暂缓项，而不是隐藏的缺口。

## PRD 就绪信号

仅当满足以下条件时，才从塑形进入 PRD：

- 来源与证据已记录；
- 主要用户、场景与价值已清晰；
- 成功指标与护栏指标已定义，或显式标注不适用；
- 业务规则与状态流转可见；
- MVP 与非目标已确认；
- 未解决问题为非关键，或已有负责人。
