# 质量门禁（Quality Gate）

## 状态模型

仅使用：

```text
draft -> review-ready -> approved
```

- `draft`：塑形或 PRD 内容不完整、未确认或被阻断。
- `review-ready`：确定性校验通过，且所有模块均已确认。
- `approved`：校验通过、状态此前已是 `review-ready`，且用户已明确批准该 PRD。

绝不把一个未通过的文档标为 `review-ready` 或 `approved`。

## 阻断条件（Blocking Conditions）

任意一条阻断项都会使严格校验失败：

- 缺少必需的 `[PRD-*]` 章节标记；
- 残留未解决的 `[待确认]`、`[TO CONFIRM]`、`[冲突]` 或 `[CONFLICT]`；
- 占位符 token，如 `{{...}}`；
- 重复的 `FR-*`、`NFR-*` 或 `AC-*` ID；
- 功能需求缺少优先级、目标/场景/来源追踪、业务规则或验收标准；
- 验收标准未关联到某个需求；
- NFR 没有可量化目标，也没有显式的"不适用"理由；
- 预期做 AI 辅助开发，却缺少 Vibe Coding 意图；
- 定稿时 `product_profile` 仍为 `auto`（必须先确认一个具体画像）；
- 画像专项阻断项，如缺少 B2B 权限矩阵、B2C 隐私/漏斗处理、AI/Data 评估与人工兜底，或通用软件的平台/运维覆盖；
- 存在阻断性的评审行动项；
- 引用了某证据 ID，但它不在 `inputs/source-notes.md` 中；
- 装配后的 PRD 在最近一次受控装配之后被手动改动；
- 校验未通过却把状态置为 `approved`。

## 评分

校验器把下列加权项累加成 **115** 的原始分，再归一化到 0–100（`round(raw * 100 / 115)`）。报告的分数与通过线均基于归一化后的 0–100 标度。

| 评分领域 | 分值 |
|---|---:|
| 必需 PRD 结构 | 20 |
| 证据与可追溯性 | 20 |
| 功能需求质量 | 25 |
| NFR 与可量化目标 | 10 |
| 流程、异常、数据、权限与集成覆盖 | 10 |
| 指标、发布、验收、风险与决策 | 10 |
| Vibe Coding 意图与评审就绪度 | 10 |
| 产品画像门禁 | 5 |
| 文档卫生与受控状态 | 5 |

通过要求：

- 归一化分数 >= 85；
- 零阻断项；
- 所有计划内模块均已确认；
- 无装配 hash 冲突。

警告本身不阻断，但在推进状态前要逐条解释每一个被接受的警告。

## 语义评审（Semantic Review）

确定性校验无法证明一个产品决策是正确的。定稿前，人工评审：

- 在发现阶段，最高风险的假设是被验证（或显式接受为风险）了，还是只是假定？
- 证据是否支撑问题与优先级？
- 所提方案是不是达成目标的最小连贯方式？
- 重要的备选方案与权衡是否可见？
- 工程能否在不靠猜测产品策略的情况下实现？
- QA 能否推导出正常、备选、异常、边界与权限的测试？
- `[PRD-METRICS]` 是否构成一棵真正的指标树（北极星 + 输入指标 + 追踪到目标的护栏）并配有实验/验证计划，而不是一串孤立数字？
- 这些指标是否能在不损害护栏的前提下证明成功？
- 灰度、回滚、运维与归属是否现实可行？

把语义评审发现记录到 `prd/quality-report.md`，置于确定性报告之后。
