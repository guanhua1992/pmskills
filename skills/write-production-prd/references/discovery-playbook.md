# 发现手册（Discovery Playbook）

在 `00-discovery` 阶段使用，位于 intake 与价值判断之前。目标是**生成证据**，而不只是记录证据：把产品想法所依赖的假设摆出来，验证其中风险最高的，并在投入 PRD 之前决定继续（proceed）、转向（pivot）还是否决（kill）。发现阶段闭合了一个回路：一个发现期假设（`ASM-*`）会"毕业"成一个生产指标（`MET-*`），以及验证它的实验（`EXP-*`）。

只有当某个决策依赖某个指标时，它才该进 PRD。如果某个数字怎么变都不改变任何决策，砍掉它。

## 假设分类

强制在每一类里至少逼出一条假设。多数失败的产品都死在前两类上。

- **需求性（Desirability）**：用户真的有这个问题、真的想要这个方案吗？
- **商业性（Viability）**：它在业务/战略上说得通吗（收入、成本、契合）？
- **可行性（Feasibility）**：在我们的技术与平台限制内能造出来吗？
- **可用性（Usability）**：目标用户真的能用它把事办成吗？

在 `shaping/00-discovery.md` 中把每条记为 `ASM-*`。

## 风险排序

先验证那个**最不确定、且一旦错了代价最大**的假设——而不是最容易验证的那个。

```
风险 = 不确定性 × 一旦错了的代价
```

可放置一旁：低不确定性或低代价的假设，可以连同负责人一起显式"接受为风险"，而不必去验证。

## 验证方法选择

| 假设类型 | 廉价的初探 | 更强的证据 |
|---|---|---|
| 需求性 | 用户访谈、工单复盘 | 行为数据、假门测试（fake-door）/落地页测试 |
| 商业性 | 定价对话、单位经济测算 | 意向书（LOI）、付费试点 |
| 可行性 | 技术预研（spike）/原型 | 压力测试或集成测试 |
| 可用性 | 概念走查 | 在原型上做有主持人的可用性测试 |

选择那个仍能**证伪**该假设的最便宜方法。一个不可能失败的测试不算验证。

## 证据强度阶梯

把每个发现映射到证据账本的 `confidence` 字段：

```
观点 / 用户口述   <  行为观察
<  行为数据 / 分析   <  受控实验
```

当假设是高风险时，不要在没有行为佐证的情况下，把一句访谈引言提升为"已验证的事实"。

## 发现决策门禁

带着一个显式结论退出发现阶段：

- **Proceed 继续**：最高风险假设已验证，或已连同负责人一起接受为已知风险。
- **Pivot 转向**：证据证伪了假设；重新框定问题或方案，并重新进入发现。
- **Kill 否决**：证据表明问题不真实、无价值或不可行——停下并明说。

把每一个仍未关闭的假设带入 `shaping/05-scope-and-version.md` 的待解决问题，附负责人与解决条件。

## 否决标准（Kill Criteria）

在下列情况下建议 `kill`（而不是悄悄缩小范围）：

- 核心需求性假设被证伪，且没有值得解决的相邻问题；
- 商业性为负，且在当前战略下无法转正；
- 唯一可行的版本已经无法解决主要场景。
