# 指标手册（Metrics Playbook）

在填写 `[PRD-METRICS]` 时使用。目标是一套**能证明产品奏效、且不会悄悄损害别处**的指标体系——而不是一串要盯着看的数字。指标闭合了发现阶段打开的回路：一个发现期假设（`ASM-*`）"毕业"成一个生产指标（`MET-*`），以及证实它的实验（`EXP-*`）。

只有当某个决策依赖某个指标时，它才属于 PRD。如果这个数字怎么变都不改变任何决策，砍掉它。

## 指标树

为每个主要目标（`GOAL-*`）建一棵树，而不是一张平铺清单：

```
北极星 (MET-001)            ← 代表"已交付价值"的那一个数字
├── 输入指标 (MET-002)      ← 产品能直接拉动的杠杆
├── 输入指标 (MET-003)
└── 护栏 (GUARD-001)        ← 在我们推动输入指标时，绝不能恶化的东西
```

- **北极星（North-star）**：滞后指标、与价值对齐、单凭自身难以被刷。每条产品线一个。
- **输入指标（Input）**：领先指标、可控、是这个版本真正会改变的东西。
- **护栏（Guardrail）**：成功不得破坏的东西——延迟、错误率、流失、成本、投诉率、信任/安全。每推动一个输入指标，至少配一个护栏。

给每个指标一个稳定 ID、一个 `追踪至 GOAL-*`、一个基线、一个目标值与一个观测窗口。只有在尚无数据时，基线才允许写 `[待确认]`——并显式说明。

## 选择北极星

用以下标准检验候选：

- 它是否只在用户获得真实价值时才变动？（而非虚荣的量）
- 团队这个周期真的能影响它吗？
- 它是否难以在不交付价值的情况下被注水？

常见形态：参与型产品 → HEART（Happiness 愉悦、Engagement 参与、Adoption 采用、Retention 留存、Task-success 任务成功）；交易型/B2C → 瓶颈步骤的漏斗转化；B2B/工作流 → 任务完成 + 价值实现时间（time-to-value）；AI/Data → 任务成功率 + 人工纠偏率 + 每任务成本/延迟。

## 埋点契约

每个 `MET-*` 都要可实现，因此要写明：

| 字段 | 示例 |
|---|---|
| 事件 / 来源 | `checkout_completed` 事件；或某张表/查询 |
| 单位与粒度 | 每用户、每会话、每日 |
| 窗口 | 滚动 7 天 |
| 分群 | 新 vs 老、套餐层级 |
| 负责人 | 谁来盯 |

如果工程无法据此推导出该事件，它就不算完成。

## 实验 / 验证计划

把"这个版本将如何被**证明**"定义为 `EXP-*`：

| 字段 | 必须答复 |
|---|---|
| 假设 | "若上线 X，则 MET-002 提升 ≥Y%，且 GUARD-001 不恶化。" |
| 方法 | A/B 测试、留出对照（holdout）、分阶段灰度、带对照的前后对比 |
| 主判定指标 | 由哪个 `MET-*` 决定成败 |
| 护栏阈值 | 触发停止/回滚的 `GUARD-*` 取值 |
| 最小可检测效应与时长 | 让实验真的能下结论 |
| 决策规则 | 推全 / 迭代 / 回滚，事先定好 |

一个无法判断成败的版本，不算 review-ready。当无法做受控实验时（如合规、样本量太小），改为陈述定性的成功信号及其判定人——不要留白。

## 反模式

- 虚荣指标（原始 PV、总注册数）且不挂任何决策。
- 输入指标没有护栏——靠悄悄搞垮另一个数字来改善一个数字。
- 目标值没有基线、也没有建立基线的计划。
- "成功以后再定"。"以后"就是你已经无法低成本改设计的时候。
