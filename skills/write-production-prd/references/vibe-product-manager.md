# Vibe 产品经理（Vibe Product Manager）

当 PRD 将用于指导 AI 辅助编码时使用。Vibe Coding 并不去除产品工作；恰恰相反，它让产品意图更重要——因为代码被生产出来的速度，比纠正决策的速度更快。

## 先有意图，再有实现

在任何交给 AI 编码之前，先定义：

- **产品目标**：系统必须创造什么样的用户/业务进展。
- **用户与场景**：谁用、何时用、为何用。
- **系统边界**：系统拥有什么、不拥有什么。
- **模块**：主要产品或系统模块，及各自职责。
- **输入**：用户输入、系统输入、数据源、事件、文件、权限。
- **输出**：UI 结果、数据变更、API 响应、事件、通知、报表。
- **约束**：策略、平台、兼容性、隐私、性能、成本、设计。
- **完成标准**：可观察的、能证明"已完成"的条件。
- **非决策项**：AI 未经产品确认不得擅自决定的话题。

## Vibe Coding 规格契约

当预期会做 AI 辅助开发时，每份最终 PRD 都应包含 `[PRD-VIBE-SPEC]`。

最小内容：

| 项 | 必须答复 |
|---|---|
| 构建意图 | 应该构建什么产品能力？ |
| 用户 | 谁使用、谁被排除？ |
| 模块 | 存在哪些模块/组件，各自拥有什么？ |
| 输入 | 什么数据或事件进入系统？ |
| 输出 | 系统应产出或改变什么？ |
| 边界 | 什么显式地在系统之外？ |
| 完成标准 | 什么可观察的测试能证明完成？ |
| AI 非决策项 | AI 在决定前必须先问什么？ |

## 迭代规则

- 生成小模块、逐模块评审，再继续。
- 把不清晰的业务规则当作阻断项，而不是实现细节。
- 让 prompt 与规格绑定到稳定的 `FR-*`、`BR-*`、`NFR-*` 与 `AC-*` ID。
- 在让 AI 写代码之前，把缺失的规则补回 PRD。
- 不要让 AI 去填补产品策略、权限、定价、合规或数据留存的缺口。

## 交接质量

只有当工程师或 AI Agent 能回答下列问题时，PRD 才算可供 Vibe Coding：

- 要构建什么；
- 系统从哪开始、到哪结束；
- 数据如何流动；
- 每个角色如何表现；
- 错误与空状态下会发生什么；
- 成功长什么样；
- 哪些决策未经产品负责人不得做出。
